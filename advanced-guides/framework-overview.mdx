MMAction2 is structured around four cooperating layers: datasets that load and index raw video data, pipelines that transform each sample into a tensor, models that process batches and produce predictions, and evaluators that aggregate metrics across the dataset. Each layer is independently configurable through Python dicts, and the MMEngine `Runner` orchestrates all four during training and evaluation.

## High-level architecture

The diagram below shows how the four layers connect at runtime.

<CardGroup cols={2}>
  <Card title="Datasets" icon="database">
    `VideoDataset`, `RawframeDataset`, `AVADataset`, and others load annotation files and produce raw data dicts for each sample.
  </Card>
  <Card title="Pipelines" icon="arrows-split-up-and-left">
    A sequence of `BaseTransform` subclasses (e.g. `DecordInit`, `SampleFrames`, `Resize`, `PackActionInputs`) converts each raw dict into a packed tensor sample.
  </Card>
  <Card title="Models" icon="brain">
    A `BaseRecognizer` (e.g. `Recognizer3D`) owns a `backbone`, optional `neck`, and `cls_head`. It exposes `loss`, `predict`, and `tensor` forward modes.
  </Card>
  <Card title="Evaluators" icon="chart-bar">
    Metrics such as `AccTopK` inherit from MMEngine's `BaseMetric`. They accumulate per-sample results and compute the final score at epoch end.
  </Card>
</CardGroup>

## The MMEngine runner lifecycle

`Runner` from `mmengine.runner` is the recommended entry point for training and evaluation. It builds every component from config dicts and drives the train/val/test loops.

```python
from mmengine.runner import Runner

runner = Runner(
    model=model_cfg,
    work_dir='./work_dirs/my_experiment',
    train_dataloader=train_dataloader_cfg,
    train_cfg=dict(type='EpochBasedTrainLoop', max_epochs=100, val_interval=1),
    val_dataloader=val_dataloader_cfg,
    val_cfg=dict(type='ValLoop'),
    optim_wrapper=dict(optimizer=dict(type='SGD', lr=0.01, momentum=0.9)),
    val_evaluator=[dict(type='AccMetric')],
    default_scope='mmaction'
)
runner.train()
```

Inside each iteration, `Runner` calls the model's unified `forward` method with the appropriate mode string:

<Steps>
  <Step title="train_step">
    The runner calls `model.forward(inputs, data_samples, mode='loss')`, which returns a dict of scalar tensors (e.g. `{'loss_cls': tensor(0.68)}`). Gradients are computed and the optimizer steps.
  </Step>
  <Step title="val_step / test_step">
    The runner calls `model.forward(inputs, data_samples, mode='predict')`, which returns a list of `ActionDataSample` objects with `pred_scores` and `pred_label` fields populated. These are forwarded to the metric's `process` method.
  </Step>
  <Step title="Metric computation">
    At the end of each validation epoch, the runner calls `metric.compute_metrics(metric.results)` to produce the final evaluation dict (e.g. `{'top1_acc': 0.756, 'top5_acc': 0.932}`).
  </Step>
</Steps>

<Note>
`mode='tensor'` is also accepted by `BaseRecognizer.forward` and returns raw features without any post-processing, useful for feature extraction or visualization.
</Note>

## Registry system

MMAction2 extends MMEngine's registry mechanism. Every component type has a named registry defined in `mmaction/registry.py`. Registries form a parent–child hierarchy: MMAction2's registries inherit from MMEngine's root registries, so MMEngine-built components (e.g. standard optimizers) are automatically available.

```python
# mmaction/registry.py (abridged)
from mmengine.registry import Registry

MODELS = Registry(
    'model', parent=MMENGINE_MODELS, locations=['mmaction.models'])

DATASETS = Registry(
    'dataset', parent=MMENGINE_DATASETS, locations=['mmaction.datasets'])

TRANSFORMS = Registry(
    'transform', parent=MMENGINE_TRANSFORMS,
    locations=['mmaction.datasets.transforms'])

METRICS = Registry(
    'metric', parent=MMENGINE_METRICS, locations=['mmaction.evaluation'])
```

The `locations` argument tells the registry which Python packages to scan for `@REGISTRY.register_module()` decorators. When you call `MODELS.build(cfg)`, the registry looks up the class by the `type` key in `cfg`, instantiates it, and passes the remaining keys as constructor arguments.

### How configs map to Python objects

```python
from mmaction.registry import MODELS

backbone_cfg = dict(
    type='ResNet3d',
    pretrained=None,
    depth=50,
    conv1_kernel=(5, 7, 7),
    conv1_stride_t=2,
    pool1_stride_t=2,
    inflate=(0, 0, 1, 1),
)

backbone = MODELS.build(backbone_cfg)
# equivalent to: ResNet3d(depth=50, conv1_kernel=..., ...)
```

Any class decorated with `@MODELS.register_module()` (or whose module is listed in `locations`) becomes available this way. This means config files are a direct serialization of the constructor call graph.

## Dataflow: raw video to ActionDataSample

The following walk-through traces a single training sample from disk to loss computation.

<Steps>
  <Step title="Annotation loading">
    `VideoDataset.load_data_list()` reads the annotation file and builds a list of dicts:

    ```python
    # kinetics_train.txt:  some/path/video.mp4 42
    data_list.append(dict(filename='some/path/video.mp4', label=42))
    ```
  </Step>
  <Step title="Dataset __getitem__">
    `BaseDataset.__getitem__(idx)` calls `get_data_info(idx)` to retrieve the raw dict, then passes it through the configured `pipeline` (a `Compose` of transforms).
  </Step>
  <Step title="Pipeline execution">
    Each transform receives and returns a `dict`. The chain is:

    ```python
    train_pipeline = [
        dict(type='DecordInit'),
        dict(type='SampleFrames', clip_len=32, frame_interval=2, num_clips=1),
        dict(type='DecordDecode'),
        dict(type='Resize', scale=(-1, 256)),
        dict(type='RandomResizedCrop'),
        dict(type='Resize', scale=(224, 224), keep_ratio=False),
        dict(type='Flip', flip_ratio=0.5),
        dict(type='FormatShape', input_format='NCTHW'),
        dict(type='PackActionInputs'),
    ]
    ```

    After `PackActionInputs`, the dict contains two keys:
    - `inputs`: a `torch.Tensor` of shape `(num_clips, C, T, H, W)`
    - `data_samples`: an `ActionDataSample` carrying `gt_label` and metainfo
  </Step>
  <Step title="DataLoader collation">
    The DataLoader stacks individual samples into a batch. `inputs` becomes a list of tensors; `data_samples` becomes a list of `ActionDataSample` objects.
  </Step>
  <Step title="Data preprocessing">
    `ActionDataPreprocessor.forward(data, training=True)` stacks the list of input tensors into a single batched tensor and applies mean/std normalization on GPU.
  </Step>
  <Step title="Model forward (loss mode)">
    `BaseRecognizer.forward(inputs, data_samples, mode='loss')`:

    1. `extract_feat` reshapes `[N, num_clips, C, T, H, W]` to `[N*num_clips, C, T, H, W]` and passes through `backbone` (and optionally `neck`).
    2. `cls_head.loss(feats, data_samples)` computes classification scores and applies the loss function, returning `{'loss_cls': tensor}`.
  </Step>
</Steps>

## Key abstractions

<AccordionGroup>
  <Accordion title="BaseRecognizer">
    Defined in `mmaction/models/recognizers/base.py`. Inherits from MMEngine's `BaseModel`. Provides the unified `forward(inputs, data_samples, mode)` entry point and delegates to `extract_feat`, `cls_head.loss`, and `cls_head.predict`. Concrete subclasses (`Recognizer2D`, `Recognizer3D`, `RecognizerGCN`) implement `extract_feat` for their specific input shapes.
  </Accordion>
  <Accordion title="BaseActionDataset">
    Defined in `mmaction/datasets/base.py`. Inherits from MMEngine's `BaseDataset`. Subclasses only need to implement `load_data_list()` — the `__getitem__` loop, pipeline application, and data prefetching are all inherited.
  </Accordion>
  <Accordion title="ActionDataSample">
    A structured data container from `mmaction/structures`. It holds `gt_label` (ground truth), `pred_scores` (raw logits), `pred_label` (argmax prediction), and arbitrary metainfo (e.g. `img_shape`, `num_clips`, `clip_len`). It is returned by both the pipeline (`PackActionInputs`) and the model's predict mode.
  </Accordion>
  <Accordion title="BaseHead">
    Defined in `mmaction/models/heads/base.py`. Implements `loss(feats, data_samples)` and `predict(feats, data_samples)` using the abstract `forward(x)`. The `loss_by_feat` method handles label smoothing and top-k accuracy logging. `predict_by_feat` handles multi-clip score averaging via `average_clip`.
  </Accordion>
</AccordionGroup>

## Initializing the scope

Before using any registry in a standalone script (outside the `Runner`), call `register_all_modules` to ensure all modules are registered under the `mmaction` scope:

```python
from mmaction.utils import register_all_modules

register_all_modules(init_default_scope=True)
```

This is not required when using `Runner` with `default_scope='mmaction'`, as the runner handles scope initialization automatically.
